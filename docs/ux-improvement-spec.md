## 1. 개요

### 목적
서비스의 리텐션 및 일상적 활용도를 높이고, 모바일 포함 전 환경에서의 사용 편의성을 개선한다.

### 핵심 방향
- **트래커리어의 강점 유지**: 캘린더 기반 촘촘한 일정 관리
- **잡트래커 참고**: 지원 현황 한눈에 보기 대시보드 방식 도입
- **진입 장벽 낮추기**: 자주 쓰는 액션의 클릭 수 감소
- **재방문 동기 부여**: 알림으로 리텐션 확보

## 2. 현황 분석

### 2-1. 현재 레이아웃 구조

```
/(public)/page.tsx
├── 비로그인: 공고 URL 입력 + 랜딩 문구
└── 로그인: flex-row 3열 레이아웃
    ├── 왼쪽 영역 (flex-1): JobPostingInput + CalendarView (2주 뷰)
    └── 오른쪽 영역 (w-80 고정): Sidebar
        ├── 탭: 진행 중 / 종료
        ├── 서브탭: 할 일 / 완료 (진행 중) | 최종 합격 / 불합격 (종료)
        └── 필터: 전체 / 서류 / 과제 / 시험 / 면접
```

### 2-2. 반응형 현황

| 컴포넌트 | 반응형 적용 | 주요 문제 |
|---|---|---|
| 메인 레이아웃 | ❌ | flex-row 고정, 모바일에서 가로 압축 |
| Sidebar | ❌ | w-80 고정, 모바일 화면 잠식 |
| CalendarView | ❌ | 7열 고정 그리드, 모바일 스크롤 불가 |
| CalendarCell | ❌ | min-h 고정 |
| JobPostingInput | ❌ | 고정 너비 |
| 모달 전반 | ❌ | 고정 너비 |

### 2-3. Tailwind 커스텀 Breakpoint

```ts
// tailwind.config.ts
screen: {
  mobile:  '320px',
  tablet:  '600px',
  laptop:  '1200px',
  desktop: '1536px',
}
```

### 2-4. 상태 변경 흐름 (현재)

```
공고 카드 클릭 → 상태 버튼(진행 중▼) 클릭
  → StageUpdateMenu 팝업 (완료 / 합격 / 불합격 / 되돌리기)
    → "완료" 선택 → StageCompleteMenu (발표 일정 입력)
    → "합격" 선택 → StagePassMenu (최종 합격 처리)
```

> 단순 "완료" 처리까지 최소 3번 클릭 필요

### 3-1. 알림 기능 (리텐션)

**목적**: 서비스 재방문 동기를 만들어 리텐션을 높인다.

**발송 시나리오**

| 트리거 | 발송 타이밍 | 메시지 예시 |
|---|---|---|
| 공고 마감 임박 | D-3, D-1 | "OO 기업 서류 마감 D-1이에요. 준비 잘 되고 있나요?" |
| 전형 일정 당일 | 당일 오전 9시 | "오늘 OO 기업 면접 일정이 있어요. 화이팅!" |
| 결과 발표 예정일 | 발표 예정 당일 오전 | "OO 기업 결과 발표일이에요. 확인 후 상태를 업데이트해주세요." |
| 미응답 공고 | 공고 등록 후 7일 이상 상태 미변경 | "OO 기업 지원 상태를 오랫동안 업데이트하지 않으셨어요." |
| 주간 요약 | 매주 월요일 오전 | "이번 주 면접 2건, 결과 대기 1건이 있어요." |

**구현 방향**

- 채널: 브라우저 Web Push (PWA) + 이메일 (선택)
- 설정: SettingModal에 알림 유형별 ON/OFF 토글 추가
- 권한 요청: 최초 로그인 후 온보딩 플로우에서 안내

**관련 파일**
- `src/components/common/modal/SettingModal.tsx` - 알림 설정 UI 추가
- `src/components/common/modal/OnboardingModal.tsx` - 알림 권한 요청 추가
- `public/sw.js` (신규) - Service Worker 등록

### 3-2. 지원 현황 대시보드 (잡트래커 참고)

**목적**: 지원 상태별로 모아서 보는 뷰를 제공해 전체 현황을 한눈에 파악하게 한다.

**현황 문제**
- Sidebar는 공고를 "가장 최근 전형 단계" 기준으로 1개씩만 표시
- 전체 공고 수와 단계별 현황을 집계해서 보여주는 화면 없음

**개선 방향: 대시보드 뷰 추가**

```
[대시보드 뷰 레이아웃]

┌─────────────────────────────────────────────┐
│  지원 현황 요약 카드                          │
│  서류 접수 (3)  |  전형 진행 중 (2)  |  결과 대기 (1)  │
│  최종 합격 (1)  |  불합격 (5)                │
└─────────────────────────────────────────────┘

┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  서류 접수   │  │  전형 진행 중 │  │  결과 대기   │
│─────────────│  │─────────────│  │─────────────│
│ OO 기업 [서류]│  │ △△ 기업 [면접]│  │ □□ 기업     │
│ D-3         │  │ D-day       │  │ 발표 D+1    │
└─────────────┘  └─────────────┘  └─────────────┘
```

**구현 방향**
- 메인 레이아웃에 캘린더 뷰 / 대시보드 뷰 전환 탭 추가
- 칸반 스타일 컬럼 (상태별 카드 묶음) 또는 테이블 형태
- 상태 컬럼: `서류 접수` → `전형 진행 중` → `결과 대기` → `최종 합격 / 불합격`

**관련 파일**
- `src/app/(public)/page.tsx` - 뷰 전환 탭 추가
- `src/components/dashboard/DashboardView.tsx` (신규) - 대시보드 컴포넌트
- `src/components/dashboard/DashboardColumn.tsx` (신규) - 칸반 컬럼

### 3-3. 상태 변경 UX 간소화

**목적**: 전형 상태 변경에 필요한 클릭 수를 줄인다.

**현황 문제**
- 상태 변경 → StageUpdateMenu → StageCompleteMenu/PassMenu 순으로 **최소 3단계**
- 결과 발표 날짜 입력이 필수로 요구되어 "일단 완료만 처리"하기 어려움

**개선 방향**

Option A. 인라인 상태 변경 (드롭다운 직접 노출)
```
[진행 중 ▼]  →  드롭다운 즉시 표시
                ├── ✅ 완료 처리 (클릭 1회로 완료, 발표일은 나중에 입력)
                ├── 🏆 최종 합격
                ├── ❌ 불합격
                └── ↩ 되돌리기
```

Option B. 스와이프 액션 (모바일)
- 카드를 오른쪽으로 스와이프 → "완료" 처리
- 카드를 왼쪽으로 스와이프 → "불합격" 처리

**권장**: Option A (데스크톱/모바일 공통 적용) + 발표일 입력은 선택사항으로 변경

**관련 파일**
- `src/components/stage/UpdateMenu.tsx` - 메뉴 구조 개선
- `src/components/stage/CompleteMenu.tsx` - 발표일 선택 optional 처리
- `src/components/sidebar/Sidebar.tsx` - 인라인 상태 변경 적용

### 3-4. 공고별 할일 관리

**목적**: 공고 준비에 필요한 일들을 공고에 묶어서 관리할 수 있게 한다.

**현황 문제**
- 기존 Todo는 날짜 기반으로만 캘린더에 표시
- "OO 기업 자기소개서 작성", "포트폴리오 준비" 같이 특정 공고에 연결된 할일 관리 불가

**개선 방향**

```
공고 상세 모달 (JobPostingDetailModal)
├── 기본 정보 (회사명, 공고명, 마감일)
├── 전형 단계 타임라인
└── [공고별 할일] 탭 (신규)
    ├── + 할일 추가
    ├── [ ] 자기소개서 1차 작성
    ├── [x] 직무 기술서 분석
    └── [ ] 모의 면접 연습 (면접 단계에서 자동 추천?)
```

**데이터 모델 (백엔드 협의 필요)**
```ts
type JobPostingTodo = {
  id: number;
  jobPostingId: number;   // 공고 연결
  stageId?: number;       // 특정 전형 단계 연결 (선택)
  content: string;
  isDone: boolean;
  createdAt: string;
};
```

**구현 방향**
- 기존 `CreateTodoModal` / `EditTodoModal` 재활용 또는 확장
- 공고 상세 모달에 할일 탭 추가
- Sidebar의 "할 일" 탭에서 공고별 묶음으로 표시하는 뷰도 고려

**관련 파일**
- `src/components/jobPosting/JobPostingDetailModal.tsx` - 할일 탭 추가
- `src/components/todo/CreateTodoModal.tsx` - jobPostingId 연결 지원
- `src/api/todo.ts` - jobPostingId 기반 API 추가

### 3-5. 캘린더 현재 인지 강화

**목적**: 사용자가 캘린더에서 "오늘"을 즉각적으로 인식할 수 있게 한다.

**현황 문제**
- "오늘로 이동하기" 버튼은 있으나 색상이 `text-disabled`로 눈에 띄지 않음
- 오늘 날짜 셀에 대한 시각적 강조가 부족할 수 있음
- 현재 주차인지 미래/과거인지 인지가 어려움

**개선 방향**

1. **오늘 날짜 강조**
   - 오늘 날짜 숫자에 Primary 색상 원형 배경 추가
   - `today` prop이 이미 `CalendarCell`에 전달되고 있으므로 CSS 조건 추가만 필요

2. **헤더 날짜 표시 개선**
   - 현재: `3월 2주차` → 개선: `3월 2주차 (3.10 - 3.16)` 기간 날짜 병기

3. **오늘로 이동 버튼 강조**
   - 현재 주가 아닐 때: Primary 색상으로 눈에 띄게 표시
   - 현재 주일 때: 비활성화 처리 (`text-disabled`, disabled)

4. **오늘 기준 D-day 표시**
   - 마감/발표 일정에 D-day 뱃지 추가 (이미 Sidebar에서는 `getSummaryStr`로 구현됨)
   - CalendarCell 내 일정 카드에도 동일 표시

**관련 파일**
- `src/components/calendar/CalendarView.tsx` - 헤더 및 오늘 이동 버튼 개선
- `src/components/calendar/CalendarCell.tsx` - 오늘 강조 스타일 추가

### 3-6. 월간 캘린더 뷰

**목적**: 2주 뷰만으로는 한 달 전체 일정 흐름을 파악하기 어려워 월간 뷰를 추가한다.

**현황 문제**
- 현재 CalendarView는 2주(14일) 고정 뷰
- 다음 달 마감 공고, 한 달 단위 취준 계획을 한눈에 보기 어려움
- 취업 준비 특성상 월 단위 일정 파악이 중요 (서류 마감, 면접 등이 1~2개월에 걸쳐 분포)

**개선 방향: 캘린더 뷰 전환 탭**

```
[2주 뷰] [월간 뷰]  ← 헤더 영역에 토글 탭 추가

월간 뷰 레이아웃 (7열 x 4~6행):
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  월  │  화  │  수  │  목  │  금  │  토  │  일  │
├─────┼─────┼─────┼─────┼─────┼─────┼─────┤
│     │     │     │     │  1  │  2  │  3  │
│     │     │     │     │[서류]│     │     │
├─────┼─────┼─────┼─────┼─────┼─────┼─────┤
│  4  │  5  │  6  │  7  │  8  │  9  │ 10  │
│[면접]│     │     │     │     │     │     │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┘
```

**2주 뷰 vs 월간 뷰 비교**

| 항목 | 2주 뷰 (현재) | 월간 뷰 (추가) |
|---|---|---|
| 표시 기간 | 14일 | 28~35일 |
| 셀 높이 | 여유 있음 | 압축 필요 |
| 일정 표시 | 상세 (할일 + 일정) | 요약 (점/뱃지 형태) |
| 주 용도 | 이번 주 할일 관리 | 한 달 일정 조망 |

**구현 포인트**

- 뷰 상태: `'2week' | 'month'` 토글 (CalendarView 내부 state)
- 월간 셀은 일정이 많을 때 `+N개 더` 말줄임 처리
- 월간 뷰에서 날짜 클릭 → 해당 날짜 기준 2주 뷰로 전환 (드릴다운)
- 데이터 패치 범위: 월 단위로 `startDate` / `endDate` 변경
- 모바일: 셀이 좁으므로 일정을 점(dot)으로만 표시 → 날짜 탭 시 하단 시트로 목록 표시

**관련 파일**
- `src/components/calendar/CalendarView.tsx` - 뷰 전환 탭 + 월간 뷰 렌더링 분기
- `src/components/calendar/MonthlyCalendarView.tsx` (신규) - 월간 뷰 컴포넌트
- `src/components/calendar/MonthlyCalendarCell.tsx` (신규) - 월간 셀 (압축 표시)

### 3-7. 드래그앤드롭

**목적**: 캘린더의 할일/일정을 드래그앤드롭으로 날짜 이동할 수 있게 한다.

**적용 범위**
- CalendarView 내 Todo 아이템의 날짜 간 이동
- (심화) 대시보드 뷰에서 공고 카드의 칸반 컬럼 간 이동으로 상태 변경

**구현 라이브러리 검토**

| 라이브러리 | 특징 | 모바일 지원 |
|---|---|---|
| `@dnd-kit/core` | 가볍고 접근성 우수, 터치 지원 | ✅ |
| `react-beautiful-dnd` | 직관적 API, 유지보수 종료 | ⚠️ |
| `react-dnd` | 유연하지만 무거움 | ⚠️ |

**권장**: `@dnd-kit/core` (모바일 터치 지원, 번들 크기 작음)

**구현 포인트**
- 날짜 이동 시 API `PATCH /todos/:id` 호출하여 `date` 변경
- 드래그 중 목표 날짜 셀 하이라이트 표시
- 모바일에서는 길게 누르기(long press)로 드래그 활성화

**관련 파일**
- `src/components/calendar/CalendarView.tsx` - DndContext 적용
- `src/components/calendar/CalendarCell.tsx` - Droppable 적용
- `src/api/todo.ts` - date 업데이트 API 연결

### 3-8. 공휴일 표시

**목적**: 백엔드에서 전달받은 공휴일 데이터를 캘린더에 반영해 일정 계획 시 공휴일을 즉각 인식할 수 있게 한다.

**현황 문제**
- 현재 캘린더에 공휴일 표시 없음
- 공고 마감일·면접 일정 등이 공휴일과 겹치는지 사용자가 직접 확인해야 함

**데이터 구조 (백엔드 협의 기반)**

```ts
type Holiday = {
  date: string;       // "YYYY-MM-DD"
  name: string;       // "삼일절", "추석" 등
  isSubstitute?: boolean; // 대체공휴일 여부
};
```

- API 또는 캘린더 데이터 응답에 `holidays: Holiday[]` 포함 (백엔드 협의)
- 프론트에서는 `date` 기준으로 캘린더 셀과 매핑

**UI 반영 방식**

```
┌─────────────────────┐
│  3  (삼일절)  공휴일  │   ← 날짜 옆에 공휴일명 + 빨간색 텍스트
│  [OO 기업 서류]       │
└─────────────────────┘
```

| 적용 위치 | 표시 방법 |
|---|---|
| 2주 뷰 셀 헤더 | 날짜 옆에 공휴일명 (빨간색 소형 텍스트) |
| 월간 뷰 셀 | 날짜 숫자를 빨간색으로 표시 + 툴팁으로 공휴일명 |
| 모바일 | 날짜 숫자 빨간색만 표시 (텍스트 생략) |

**구현 포인트**

- 공휴일 데이터는 월 단위로 패치 시 함께 받아오는 것이 효율적
- `CalendarCell` / `MonthlyCalendarCell` 모두에 `holidays` prop 추가
- 대체공휴일(`isSubstitute: true`)은 별도 표기 고려 (`*` 또는 별도 색상)

**관련 파일**
- `src/components/calendar/CalendarView.tsx` - 공휴일 데이터 패치 및 셀에 전달
- `src/components/calendar/CalendarCell.tsx` - 공휴일 표시 처리
- `src/components/calendar/MonthlyCalendarCell.tsx` (신규) - 월간 셀 공휴일 표시
- `src/api/calendar.ts` (또는 기존 API 파일) - 공휴일 포함 응답 타입 정의

### 3-9. 면접 회고 & 메모 UX 강화

**목적**: 기존 `JobPosting.memo` API를 그대로 유지하면서 사용자가 면접 회고와 메모를 더 편하게 작성할 수 있도록 UX를 개선한다.

**현황 문제**
- 메모 수정 경로: 공고 상세 모달 → 편집 모달 열기 → 스크롤 → 메모 textarea → 저장 (너무 멀다)
- `JobPostingEditModal` 하단의 작은 textarea (min-h-19.5)로만 작성 가능
- 면접 전형이 있어도 일반 "메모"로만 표시 — 회고 맥락이 없음
- 메모가 있어도 상세 모달에서 `-`로만 표시되어 편집 의욕이 없음

---

#### A. 상세 모달 인라인 편집 (가장 임팩트 큰 개선)

**현재 흐름**
```
상세 모달(읽기전용) → [편집] 버튼 → EditModal 열기 → 메모 수정 → 저장
```

**개선 후**
```
상세 모달에서 메모 영역 클릭 → 즉시 인라인 편집 → 포커스 아웃 또는 저장 버튼으로 저장
```

구현 포인트:
- `JobPostingDetailModal`의 메모 표시 영역을 클릭하면 `<p>` → `<textarea>`로 전환
- `onBlur` 또는 별도 저장 버튼으로 `PUT /job-postings/:id` 호출 (기존 API 재사용)
- 저장 중 스피너, 저장 완료 후 원래 읽기 모드로 복귀

---

#### B. 면접 완료 시 회고 작성 유도

면접 전형을 "완료" 처리할 때 메모 작성을 자연스럽게 유도한다.

**현재**: `CompleteMenu`는 발표 예정일만 입력

**개선**: 면접(`INTERVIEW`) 단계 완료 시 "면접 회고를 남겨두면 다음에 도움이 돼요" 안내 + 메모 textarea 추가

```
┌──────────────────────────────────┐
│  결과 발표 예정일                  │
│  [날짜 선택]     [별도 안내 없음]   │
│                                  │
│  면접 회고 (선택)                  │
│  ┌────────────────────────────┐  │
│  │ 분위기, 예상치 못한 질문,   │  │
│  │ 개선점 등을 자유롭게 기록   │  │
│  └────────────────────────────┘  │
│                     [완료 처리]   │
└──────────────────────────────────┘
```

구현 포인트:
- `CompleteMenu`에서 `stageType === 'INTERVIEW'`일 때 메모 textarea 조건부 렌더링
- 입력한 내용은 기존 `memo` 필드에 **append** (기존 메모에 `\n\n---\n[면접 회고 - YYYY.MM.DD]\n내용` 형식으로 추가)
- API는 `PUT /job-postings/:id` 그대로 사용

---

#### C. 회고 템플릿 제안

메모 영역에 면접 단계가 있을 때 템플릿 버튼을 제공한다.

```
┌─────────────────────────────────────────────┐
│  메모 / 회고                  [템플릿 불러오기] │
│  ┌───────────────────────────────────────┐  │
│  │ 분위기는 어땠나요?                      │  │
│  │ 예상치 못한 질문이 있었나요?             │  │
│  │ 다음에 보완할 점은?                     │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

**템플릿 내용 (INTERVIEW 단계용)**
```
## 면접 분위기
-

## 받은 질문
-

## 잘 답변한 것
-

## 보완할 점
-
```

구현 포인트:
- 템플릿은 클라이언트 상수로 관리 (API 변경 없음)
- [템플릿 불러오기] 클릭 시 기존 메모가 있으면 "기존 내용이 대체됩니다" 확인 알림
- `stageType`에 따라 다른 템플릿 제공 가능 (서류: 자소서 체크리스트, 과제: 완료 체크리스트)

---

#### D. 메모 편집 영역 UX 세부 개선

| 항목 | 현재 | 개선 |
|---|---|---|
| textarea 높이 | `min-h-19.5` 고정 | 내용에 따라 자동 확장 (`auto-resize`) |
| 글자수 | 없음 | 우측 하단에 `현재/최대` 표시 |
| 빈 상태 안내 | `-` 표시 | "클릭하여 메모 추가" 유도 문구 |
| 라벨 | "메모" 고정 | 면접 단계 있으면 "메모 / 회고"로 변경 |

## 4. 모바일 반응형 전략

### 4-1. Breakpoint 적용 기준

| Breakpoint | 사이즈 | 레이아웃 |
|---|---|---|
| `mobile` (< 600px) | 320px~ | 단열, Sidebar 드로어 |
| `tablet` (600px~) | 600px~1200px | 캘린더 축소 + 사이드 패널 |
| `laptop` (1200px~) | 1200px~ | 현재 3열 레이아웃 유지 |

### 4-2. 컴포넌트별 반응형 계획

**메인 레이아웃 (`page.tsx`)**
```
laptop 이상:  flex-row (캘린더 + 사이드바 나란히)
tablet 이하:  flex-col (캘린더 위 → 사이드바 아래)
mobile:       탭 전환 방식 (캘린더 탭 / 공고 목록 탭)
```

**Sidebar**
```
laptop 이상:  w-80 고정, 항상 표시
tablet 이하:  하단 시트 또는 오른쪽 슬라이드 드로어
mobile:       하단 플로팅 버튼 → 드로어 열기
```

**CalendarView**
```
laptop 이상:  2주 뷰 / 월간 뷰 탭 전환
tablet:       2주 뷰 (셀 축소) / 월간 뷰 (압축)
mobile:       2주 뷰 → 가로 스크롤 or 1주 뷰
              월간 뷰 → dot 표시 + 날짜 탭 시 하단 시트
```

**모달 전반**
```
laptop 이상:  중앙 고정 너비 모달
mobile:       하단 시트(bottom sheet) 방식으로 전환
```

### 4-3. 공통 적용 사항

- 터치 타깃 최소 44x44px 보장 (현재 일부 버튼 미충족)
- 폰트 크기 mobile에서 최소 14px
- 스크롤 가능한 모달에 `overscroll-contain` 적용
- iOS Safari 대응: `safe-area-inset` 고려 (`pb-safe` 등)
